1. Product thesis

An agent should be a directory.

The framework will let developers define an agent using ordinary files: Markdown instructions, YAML manifests, TypeScript tools, eval fixtures, schemas, workflows, and generated run artifacts. The first version must run entirely locally without Docker sandboxing, while preserving architectural seams for future Docker sandboxing and cloud adapters for Azure, Google Vertex / Agent Platform, and AWS.

The framework should be built for frontier coding agents to extend. Claude Code and Codex should be able to read the repository, understand the architecture, implement missing components, run tests, inspect failures, and continue iterating without human step-by-step prompting.

First proof agent: a local PDF summarization agent that compiles, runs, writes trace/events, writes a Markdown summary artifact, and passes evals without external API credentials.

2. Goals and non-goals

Primary goals

  1. Initialize an agent project.
  2. Compile an agent/ directory into a portable manifest.
  3. Run the compiled agent locally.
  4. Persist run state, events, tool calls, traces, and artifacts.
  5. Support local tools without Docker sandboxing in V1.
  6. Support workflows as deterministic orchestration files.
  7. Support simple agentic execution later, but prioritize workflow execution first.
  8. Support evals as first-class files.
  9. Include a working local PDF summarization agent.
  10. Make future Docker, Azure, Vertex, and AWS adapters obvious to implement.

Non-goals for V1

Docker sandbox execution
Azure deployment
Google Vertex / Agent Platform deployment
AWS deployment
Remote multi-tenant control plane
Visual workflow builder
Long-term semantic memory
OAuth connector framework
Human approval UI
Vector search
Subagent fan-out/fan-in runtime

3. Audience and agent-loop fit

The direct implementation audience is Claude Code, Codex, other frontier coding agents, and a human technical reviewer.

Claude’s Agent SDK describes an agent loop as a cycle where the model receives a prompt plus tools/history, decides whether to call tools, executes tools, feeds results back, repeats until no tool calls remain, then emits a final result with token usage, cost, and session ID. It also supports max turns, budget limits, permissions, hooks, subagents, and OpenTelemetry observability. This PRD’s verification loops are designed to fit that style of autonomous coding workflow. Claude Agent SDK: Agent Loop

4. Architectural principles

Agent directory as source

agent/ is source code. Runtime output belongs in .agent/.

Compile before execution

The runtime executes compiled manifests, not arbitrary loose files.

Local runtime as simulator

agent deploy --target local should be production-like, strict, durable, and addressable.

Future cloud adapters

V1 has interfaces for cloud targets, but no cloud SDK dependencies.

No Docker in V1

Run tools in-process or subprocess. Docker becomes a future executor, not a rewrite.

Tracing is core

Every run emits structured events and trace records.

Required abstractions

RuntimeAdapter
StateStore
ArtifactStore
ToolExecutor
WorkflowEngine
TraceEmitter

SandboxProvider
DeploymentTarget

Use SQLite for the local durable store. SQLite WAL mode is a good fit for local state because it improves read/write concurrency and records changes before checkpointing them back to the main database. SQLite WAL

Responsibilities

  • Read and validate agent.yaml.
  • Read instructions.md.
  • Discover tools, workflows, skills, evals, and schemas.

10. Runtime contract

The runtime must execute compiled manifests, not raw source.

Interfaces

interface RuntimeAdapter {
  startRun(input: RunInput): Promise<RunHandle>;
  resumeRun(runId: string, event: ExternalEvent): Promise<RunHandle>;
  cancelRun(runId: string): Promise<void>;
  getRun(runId: string): Promise<RunRecord>;
}

interface StateStore {
  createRun(record: CreateRunRecord): Promise<RunRecord>;
  appendEvent(runId: string, event: RunEvent): Promise<void>;
  createStep(step: CreateStepRecord): Promise<StepRecord>;
  updateStep(stepId: string, patch: Partial<StepRecord>): Promise<void>;
  listPendingSteps(): Promise<StepRecord[]>;
}

interface ArtifactStore {
  writeArtifact(runId: string, artifact: ArtifactWrite): Promise<ArtifactRecord>;
  readArtifact(artifactId: string): Promise<Buffer>;
}

interface ToolExecutor {
  execute(call: ToolCall, ctx: ToolContext): Promise<ToolResult>;
}

interface TraceEmitter {
  emit(event: TraceEvent): Promise<void>;
}

States and events

Run states:
  created, running, waiting, succeeded, failed, cancelled

Step states:
  pending, running, succeeded, failed, retrying, skipped

Required events:
  run.started, run.completed, run.failed
  manifest.loaded, instructions.loaded
  workflow.started, workflow.completed
  tool.called, tool.completed, tool.failed
  artifact.written
  eval.started, eval.completed
  policy.blocked
Verifiable condition:
Run a fake workflow with two fake tools. Success means the runtime creates a run, events are sequential, state transitions are correct, failures emit tool.failed, and success emits run.completed.

12. Workflow engine

V1 should prioritize deterministic workflows over fully autonomous model loops.

agent/workflows/summarize_pdf.workflow.ts

Workflow interface

export default defineWorkflow("summarize_pdf", async ({ input, tools, artifacts, trace }) => {
  const extracted = await tools.extract_pdf_text({ filePath: input.filePath });
  const chunks = await tools.chunk_document({ pages: extracted.pages });
  const summary = await tools.summarize_chunks({ chunks: chunks.chunks });
  const artifact = await tools.write_artifact({
    filename: "summary.md",
    content: summary.markdown
  });
  return { artifact };
});

The engine must load the default workflow, provide a typed tool proxy, emit workflow events, persist tool calls, return workflow output, and write run output JSON.

Verifiable condition:
Run a workflow that calls three tools in sequence. Success means workflow output is returned, each call is recorded, events appear in order, and the proxy prevents calling tools not listed in the manifest.

13. Tool system

Tool file shape

import { defineTool } from "@agent-directory/core";
import { z } from "zod";

export default defineTool({
  name: "write_artifact",
  description: "Write an artifact to the current run artifact directory.",
  risk: "write",
  input: z.object({
    filename: z.string(),
    content: z.string()
  }),
  async execute(input, ctx) {
    return ctx.artifacts.write({
      filename: input.filename,
      content: input.content,
      mimeType: "text/markdown"
    });
  }
});

Required metadata

name
description
risk: read | write | network | destructive
input schema
execute function

V1 execution modes

inProcess
subprocess

Default to inProcess. Docker is not implemented in V1.

Minimum policy enforcement

  • Write tools may only write inside the run artifact directory.
  • Network tools are blocked unless policy.allow_external_network = true.
  • Destructive tools fail unless explicitly allowed.
  • Unknown tools cannot be called.
  • Tool input must validate against schema.
Verifiable condition:
Test read_ok, write_artifact, network_blocked, and invalid_input. Success means expected pass/fail behavior and all calls recorded in tool_calls and events.

21. Coding-agent project instructions

Create both AGENTS.md and CLAUDE.md. Keep them short and operational.

# Agent Directory Framework

## Architecture

This is a filesystem-based agent framework. The source of truth for an agent is `agent/`. Runtime output belongs in `.agent/`.

## Commands

- `pnpm build`
- `pnpm test`
- `pnpm agent compile examples/pdf-summarizer`
- `pnpm agent run examples/pdf-summarizer --input examples/pdf-summarizer/agent/fixtures/sample.pdf`
- `pnpm agent eval examples/pdf-summarizer`

## Rules

- Do not add Docker, Azure, AWS, or GCP SDKs in V1.
- Do not require paid model credentials for tests.
- Keep local runtime deterministic.
- Add tests for every package change.
- Prefer small interfaces over framework magic.
- Do not silently ignore malformed agent files.

Claude’s docs note that persistent project rules such as CLAUDE.md are loaded into the session and re-injected across requests, whereas ordinary conversation context can be compacted as sessions grow. Keep these files short so they help rather than bloat context. Claude Agent SDK: Agent Loop

Verifiable condition:
Both files exist, include commands and V1 constraints, and neither exceeds 150 lines.
