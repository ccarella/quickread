Product Requirements Document — Revised

Agent Directory Compiler and Local Harness

A filesystem-native agent specification, compiler, and local verification harness. It is intentionally not a replacement for Azure Foundry, AWS AgentCore, Google ADK / Vertex Agent Platform, LangGraph, OpenAI Agents SDK, or Anthropic Agent SDK.

Audience: Claude Code, Codex, frontier coding agents, and human technical reviewers. Proof target: a local PDF summarization agent using the Anthropic API for integration tests.

1. Strategic clarification

This project should not compete with native agent runtimes. Its job is to define a filesystem-native authoring model, compiler, local verification harness, trace/eval/artifact conventions, and adapter contract.

The local runtime is a reference implementation for development and proof. Production adapters should target native platforms whenever possible: Azure Foundry / Azure Agent Framework, AWS AgentCore, Google ADK / Vertex Agent Platform, LangGraph, OpenAI Agents SDK, Anthropic Agent SDK, or Docker.

Product category: this is not a general-purpose agent SDK. It is an agent directory specification + compiler + local harness.

Authoring model

agent/ source files
        ↓
compiler validation
        ↓
portable manifest
        ↓
local verification harness
        ↓
native runtime adapter later

What this project must protect

  • Agent behavior is source-controlled.
  • Instructions, tools, schemas, workflows, evals, and fixtures live together.
  • Local runs produce inspectable evidence.
  • Manifest semantics remain portable across future runtime targets.
  • Native platforms do the heavy production work later.

2. Goals and non-goals

Primary goals

  1. Initialize an agent directory.
  2. Compile an agent/ directory into a portable manifest.
  3. Run the compiled manifest locally through a reference harness.
  4. Persist run state, events, tool calls, traces, and artifacts.
  5. Support local tools without Docker sandboxing in V1.
  6. Support deterministic workflow files as the first orchestration mode.
  7. Support evals as first-class files.
  8. Use the Anthropic API for the PDF summarizer integration test path.
  9. Include a working local PDF summarization proof agent.
  10. Make Docker, Azure, Vertex, AWS, and LangGraph adapter seams obvious.

Explicit non-goals for V1

3. Audience and agent-loop fit

The direct implementation audience is Claude Code, Codex, other frontier coding agents, and a human technical reviewer.

Claude’s Agent SDK describes an agent loop as a cycle where the model receives a prompt plus tools/history, decides whether to call tools, executes tools, feeds results back, repeats until no tool calls remain, then emits a final result with token usage, cost, and session ID. It also supports max turns, budget limits, permissions, hooks, subagents, and OpenTelemetry observability. This PRD’s verification loops are designed to fit that style of autonomous coding workflow. Claude Agent SDK: Agent Loop

4. Architectural principles

Agent directory as source

agent/ is source code. Runtime output belongs in .agent/.

Compile before execution

The runtime executes compiled manifests, not arbitrary loose files.

Local as reference

The local runtime proves behavior and evals. It is not the canonical production runtime.

Native backends later

Azure, AWS, Vertex, Docker, and LangGraph are adapter targets, not V1 dependencies.

No Docker in V1

Run tools in-process first. Docker becomes a future executor, not a rewrite.

Tracing is core

Every run emits structured events and trace records.

Command behavior

  • agent init creates agent/ source and a placeholder .agent/.
  • agent dev watches files, recompiles on change, and prints diagnostics. Keep thin in V1.
  • agent compile emits .agent/compiled/manifest.json, manifest.hash, prompt-bundle.md, and diagnostics.json.
  • agent run compiles if needed and executes one local task.
  • agent eval runs eval files under agent/evals/.
  • agent deploy --target local starts the local reference runtime and HTTP API.
  • agent plan --target ... validates that a compiled manifest can be mapped to the requested target and emits a deployment plan or explicit unsupported-feature diagnostics.
Verifiable condition:
Run init, compile, run, eval, local deploy, and all target plans. Success means real commands execute, unsupported targets return clear plan diagnostics, and local deploy health returns 200.

The compiler transforms filesystem source into a portable manifest.

Inputs

Outputs

Responsibilities

  • Read and validate agent.yaml.
  • Read instructions.md.
  • Discover tools, workflows, skills, evals, and schemas.
  • Validate declared default workflow exists.
  • Validate runtime, model, and policy settings.
  • Generate manifest hash and diagnostics.
  • Emit actionable warnings for unsupported future features.

Manifest shape

type CompiledManifest = {
  schemaVersion: "0.1";
  name: string;
  version: string;

  sourceRoot: string;
  manifestHash: string;
  instructions: { file: string; contentHash: string };

  runtime: {
    durability: "required" | "optional";
    sandbox: "none" | "docker" | "cloud";
    maxRunTimeMs: number;
  };

11. Local reference runtime

The local runtime is a reference implementation and production simulator. Its purpose is to validate manifests, run proof agents locally, execute evals, emit traces, prove tool/artifact behavior, and support coding-agent development loops.

Important: the local runtime is not intended to become a cloud-scale durable orchestration system. Do not add complexity that belongs in Temporal, Durable Functions, Step Functions, LangGraph, AgentCore, or Foundry.

12. Workflow engine

V1 prioritizes deterministic workflows over autonomous model loops.

agent/workflows/summarize_pdf.workflow.ts

Workflow interface

export default defineWorkflow("summarize_pdf", async ({ input, tools, model, artifacts, trace }) => {
  const extracted = await tools.extract_pdf_text({ filePath: input.filePath });
  const chunks = await tools.chunk_document({ pages: extracted.pages });
  const summary = await model.generate({
    task: "summarize_pdf_chunks",
    input: { chunks: chunks.chunks }
  });
  const artifact = await tools.write_artifact({
    filename: "summary.md",
    content: summary.text
  });
  return { artifact };
});

The engine must load the default workflow, provide a typed tool proxy, provide the configured model provider, emit workflow events, persist model/tool calls, return workflow output, and write run output JSON.

Verifiable condition:
Run a workflow that calls tools and the Anthropic provider. Success means workflow output is returned, model/tool calls are recorded, and the proxy prevents undeclared tool calls.

13. Tool system

Tool file shape

import { defineTool } from "@agent-directory/core";
import { z } from "zod";

export default defineTool({
  name: "write_artifact",
  description: "Write an artifact to the current run artifact directory.",
  risk: "write",
  input: z.object({
    filename: z.string(),
    content: z.string()
  }),
  async execute(input, ctx) {
    return ctx.artifacts.write({
      filename: input.filename,
      content: input.content,
      mimeType: "text/markdown"
    });
  }
});

Required metadata

name
description
risk: read | write | network | destructive
input schema
execute function

V1 execution modes

inProcess

Do not implement Docker in V1. Subprocess execution may be added after the proof works, but it is not required for the first success state.

Minimum policy enforcement

  • Write tools may only write inside the run artifact directory.
  • Network tools are blocked unless policy.allow_external_network = true.
  • Destructive tools fail unless explicitly allowed.
  • Unknown tools cannot be called.
  • Tool input must validate against schema.
Verifiable condition:
Test read_ok, write_artifact, network_blocked, and invalid_input. Success means expected pass/fail behavior and all calls recorded.

17. Anthropic API testing model

Use the Anthropic API for the proof agent integration tests. Unit tests should mock ModelProvider, but the end-to-end PDF proof must call Anthropic when ANTHROPIC_API_KEY is present.

Anthropic’s current model overview lists Claude Fable 5, Claude Opus 4.8, Claude Sonnet 5, and Claude Haiku 4.5, with API IDs such as claude-fable-5, claude-opus-4-8, and claude-sonnet-5. Anthropic recommends Claude Opus 4.8 for complex agentic coding and enterprise work when unsure, while Claude Sonnet 5 is positioned as a strong speed/intelligence balance. Anthropic models overview

V1 default

ANTHROPIC_MODEL=claude-sonnet-5

Allow override:

ANTHROPIC_MODEL=claude-opus-4-8
ANTHROPIC_MODEL=claude-fable-5

Provider interface

interface ModelProvider {
  generate(request: GenerateRequest): Promise<GenerateResult>;
  generateStructured<T>(request: StructuredGenerateRequest<T>): Promise<T>;
}

Anthropic provider requirements

Reads ANTHROPIC_API_KEY from environment
Reads ANTHROPIC_MODEL, default claude-sonnet-5
Uses Messages API
Supports temperature from manifest
Records usage when available
Emits model.called, model.completed, model.failed
Does not log API key
Retries only safe transient failures
Returns clear error if ANTHROPIC_API_KEY is missing

The Anthropic Messages API is the relevant API reference for implementing calls. Anthropic Messages API

Testing note

Because live model output is not guaranteed to be byte-stable, evals must assert structure and behavior rather than exact text. Use temperature: 0, but do not require byte-identical snapshots for Anthropic integration tests.

Verifiable condition:
Run the same PDF summarizer twice with temperature: 0. Success means both runs satisfy structural evals, both call Anthropic, and neither requires exact byte-stable output.

20. Native-runtime adapter architecture

Adapters should prefer native runtimes, not recreate the local runtime in the cloud.

Adapter interface

interface BackendCompiler {
  target: "local" | "docker" | "azure" | "vertex" | "aws" | "langgraph";
  plan(manifest: CompiledManifest): Promise<DeploymentPlan>;
  emit?(manifest: CompiledManifest): Promise<GeneratedBackendProject>;
}

interface DeploymentAdapter {
  target: "local" | "docker" | "azure" | "vertex" | "aws" | "langgraph";
  deploy(request: DeployRequest): Promise<DeployResult>;
  destroy?(request: DestroyRequest): Promise<void>;
  status?(request: StatusRequest): Promise<DeploymentStatus>;
}

22. Coding-agent project instructions

# Agent Directory Compiler and Local Harness

## Architecture

This project is a filesystem agent specification, compiler, and local verification harness. It is not a replacement for Azure Foundry, AWS AgentCore, Google ADK, LangGraph, OpenAI Agents SDK, or Anthropic Agent SDK.

The source of truth for an agent is `agent/`. Runtime output belongs in `.agent/`.

## Commands

- `pnpm build`
- `pnpm test`
- `pnpm agent compile examples/pdf-summarizer`
- `ANTHROPIC_API_KEY=... pnpm agent run examples/pdf-summarizer --input examples/pdf-summarizer/agent/fixtures/sample.pdf`
- `ANTHROPIC_API_KEY=... pnpm agent eval examples/pdf-summarizer`
- `pnpm agent plan --target azure examples/pdf-summarizer`

## Rules

- Do not add Docker, Azure, AWS, or GCP SDKs in V1.
- Do use Anthropic API for the proof-agent integration path.
- Do not require exact byte-stable model output from Anthropic.
- Keep local runtime deterministic except for model output.
- Add tests for every package change.
- Prefer small interfaces over framework magic.
- Do not silently ignore malformed agent files.
- Do not rebuild managed cloud runtime capabilities.

Claude’s docs note that persistent project rules such as CLAUDE.md are loaded into the session and re-injected across requests, whereas ordinary conversation context can be compacted as sessions grow. Keep these files short so they help rather than bloat context. Claude Agent SDK: Agent Loop

Verifiable condition:
Both files exist, include commands and V1 constraints, and neither exceeds 150 lines.
