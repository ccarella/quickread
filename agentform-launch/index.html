Introducing Agentform

A filesystem-native way to define, test, and run agents.

Agentform is built around one simple idea: an agent should be a directory.

Today we’re releasing the first version of Agentform, a source-first way to define agents as files, compile them into portable manifests, and prove their behavior locally before they ever touch a cloud runtime.

The idea is simple:

An agent should be a directory.

Not a dashboard object. Not a pile of SDK calls. Not a prompt hidden in a database. Not a workflow diagram that only exists inside one cloud console.

A directory.

agent/
  agent.yaml
  instructions.md
  tools/
  workflows/
  skills/
  evals/
  fixtures/

That sounds almost too ordinary. But ordinary is the point.

Files are the most durable interface we have in software. They can be read, reviewed, diffed, tested, versioned, copied, forked, and fixed. They work with Git. They work with CI. They work with coding agents. They work with humans under deadline pressure at 11:48 p.m. trying to understand why production behavior changed.

That is the bar agents need to meet.

Why we built Agentform

Most agent development still feels too slippery.

You can build something impressive quickly. That part is no longer hard. The hard part comes after the demo:

  • What exactly is the agent allowed to do?
  • Which instructions were active?
  • Which tools can it call?
  • Who changed the refund policy?
  • Did the evals pass?
  • What did the agent read?
  • Where did the output artifact come from?
  • Can I run the same thing locally before putting it in front of a customer?

These questions sound boring. They are not. They are the difference between a toy and a system.

The current agent ecosystem is rich, but fragmented. Azure Foundry, AWS AgentCore, Google ADK, LangGraph, OpenAI Agents SDK, Anthropic’s tools, Vercel Eve, and others are all moving fast. That is good. We do not need another framework trying to replace all of them.

What we need is a better source layer.

Agentform is that layer.

It gives you a filesystem specification, a compiler, a local verification harness, evals, traces, artifacts, and a path to native runtimes later.

The goal is not to compete with the clouds. The goal is to make agent behavior legible before it gets there.

What Agentform is

Agentform has three parts.

First, a convention for defining agents as files.

agent/
  agent.yaml              # manifest and runtime intent
  instructions.md         # main behavioral contract
  workflows/              # deterministic orchestration
  tools/                  # typed executable capabilities
  skills/                 # reusable natural-language procedures
  evals/                  # behavioral tests
  fixtures/               # test inputs
  schemas/                # structured handoff contracts

Second, a compiler.

The compiler reads the directory, validates it, finds tools and workflows, checks policy, produces diagnostics, and emits a portable manifest.

agent compile examples/pdf-summarizer

That manifest becomes the boundary between source and runtime.

Third, a local harness.

The local harness runs the compiled manifest on your machine. It records events, model calls, tool calls, traces, and artifacts.

agent run examples/pdf-summarizer --input examples/pdf-summarizer/fixtures/sample.pdf

After a run, you can inspect what happened:

.agent/
  compiled/
    manifest.json
  runs/
    run_abc123/
      events.jsonl
      trace.jsonl
      artifacts/
        summary.md

That is the basic promise: every run leaves evidence.

What Agentform is not

Agentform is not trying to become Azure Foundry. It is not trying to become AWS AgentCore. It is not trying to become Google ADK. It is not trying to become LangGraph. It is not trying to become a hosted agent control plane.

Those systems should do what they are good at: managed runtime, scaling, identity, observability, sandboxing, deployment, integrations, and enterprise controls.

Agentform is earlier in the lifecycle.

It is for authoring, validating, running locally, testing, and producing a clean manifest that can later target native runtimes.

The local runtime is a reference harness, not a religion.

The first proof: a PDF summarizer

The first example is deliberately simple: a PDF summarization agent.

Not because summarizing PDFs is the future of software. It is not. But it is a good test case because it touches the whole system:

  • source layout
  • manifest compilation
  • typed tools
  • workflow execution
  • model calls
  • artifact writing
  • traces
  • evals
  • local deploy

The example agent looks like this:

examples/pdf-summarizer/
  agent/
    agent.yaml
    instructions.md
    workflows/
      summarize_pdf.workflow.ts
    tools/
      extract_pdf_text.ts
      chunk_document.ts
      write_artifact.ts
    skills/
      technical-summary.md
    evals/
      basic-summary.eval.yaml
    fixtures/
      sample.pdf

The workflow is intentionally plain:

extract PDF text
chunk document
summarize chunks
write summary.md
record events and trace

The proof is not that the summary is magical. The proof is that the system is inspectable.

You can see which tool extracted the text. You can see which model was called. You can see which artifact was written. You can see whether the eval passed. You can rerun it locally.

That is the foundation we want before adding cloud deployment, subagents, Docker sandboxing, or native runtime adapters.

Designed for coding agents

One of the reasons we chose a filesystem-first design is that coding agents are good at files.

Claude Code, Codex, Cursor, and other frontier coding agents can read this structure, edit it, add tests, update tools, run evals, and make pull requests.

That matters.

A human can say:

“Add an eval that fails if the PDF summary does not include page references.”

A coding agent can add:

agent/evals/page-references.eval.yaml

Then run:

agent eval examples/pdf-summarizer

That is the loop we care about: agent behavior changed through source, tested through evals, reviewed through diffs.

This is also why we include project files like AGENTS.md and CLAUDE.md. The framework should be understandable not only to humans, but to the tools humans are now using to build software.

The commands

The first release supports the core lifecycle:

agent init
agent compile
agent run
agent eval
agent deploy --target local
agent plan --target azure
agent plan --target aws
agent plan --target vertex
agent plan --target langgraph

The plan commands are important. They do not deploy to those platforms yet. They check whether the manifest can be mapped cleanly to future targets.

That keeps us honest.

If the local manifest quietly bakes in assumptions that make Azure, AWS, Vertex, Docker, or LangGraph impossible later, we want to know early.

Why this matters

The industry does not need another agent framework for its own sake.

What it does need is a better operating model for agents.

Agents are not just code. They are code plus instructions plus tools plus state plus model behavior plus policy plus traces plus evals. If those pieces are scattered across repos, dashboards, prompts, cloud settings, and notebooks, the system becomes hard to reason about.

Putting the agent in a directory gives us a place to stand.

  • It makes the agent reviewable.
  • It makes the agent testable.
  • It makes the agent portable.
  • It makes the agent editable by humans and coding agents.
  • It makes local proof possible before cloud deployment.

That is not glamorous infrastructure. It is useful infrastructure.

And useful infrastructure is what lets the glamorous demos become real systems.

What comes next

This release is intentionally small.

The next steps are clear:

  1. Docker sandboxing for local tool execution.
  2. Azure adapter planning and deployment.
  3. LangGraph backend compilation for more complex workflows.
  4. AWS and Vertex adapter paths.
  5. Better trace inspection.
  6. Subagent support.
  7. Corpus-level examples, including batch PDF summarization.

But the first milestone is simpler:

Can an agent be a directory? Can it compile? Can it run locally? Can it produce artifacts? Can it prove behavior with evals? Can a coding agent safely extend it?

Now the answer is yes.

That is the release.

The agent is a directory.

The tool is Agentform.
