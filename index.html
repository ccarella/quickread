Research report

Filesystem Agents, Vercel Eve, and an Azure-Native Agent Framework

A technical fellow’s analysis of the filesystem-first agent pattern, what Vercel’s Eve gets right, how Microsoft’s Azure agent environment compares, and how to build an Azure-native version without merely copying Eve.

Prepared July 2, 2026. Sources include Vercel Eve documentation and launch materials, Microsoft Agent Framework documentation, and Microsoft Foundry Agent Service documentation.

1. Executive summary

Filesystem agents are an architectural pattern in which an agent is defined by a repository layout rather than assembled imperatively in application code. The agent’s instructions, tools, skills, subagents, schedules, approvals, tests, and deployment metadata live as files under a conventional directory structure. A runtime scans the directory, validates it, compiles it into an executable agent, and binds it to production infrastructure.

Vercel’s Eve is the clearest current example of this pattern. Vercel describes Eve as a filesystem-first framework for durable backend AI agents. A developer defines an agent under an agent/ directory; Eve discovers those files and compiles them into an app that runs on Vercel Functions. The Vercel documentation states that Eve uses Vercel Workflows for session persistence and resumption, Vercel Sandbox for isolated code execution, AI Gateway for model routing and fallbacks, Vercel Connect for OAuth/API-key management, and Vercel Observability for runs, token usage, and performance telemetry.1

Microsoft’s closest equivalent is not filesystem-first. It is the combination of Microsoft Agent Framework and Microsoft Foundry Agent Service. Microsoft Agent Framework combines the simpler agent abstractions associated with AutoGen and the enterprise-oriented features of Semantic Kernel: session state, type safety, middleware, telemetry, and graph-based workflows for multi-agent orchestration.2 Foundry Agent Service is the managed Azure platform for building, deploying, and scaling agents, including prompt agents and hosted agents. It supports custom code packaged as containers, managed endpoints, identity, observability, private networking, versioning, publishing, and enterprise governance.3

Bottom line: Eve is a productized, filesystem-first agent harness tightly integrated with Vercel’s infrastructure. Azure has the stronger enterprise substrate, but Microsoft has not made the filesystem itself the primary developer interface. The opportunity is to build an Azure-native filesystem agent specification and compiler that targets Microsoft Agent Framework, Foundry Agent Service, Azure Durable Functions, Azure Container Apps, Azure OpenAI / Foundry models, Azure AI Search, Entra ID, Key Vault, Application Insights, and OpenTelemetry.

2. What filesystem agents are

A filesystem agent treats a folder as the source of truth for an agent. This sounds small, but it is a serious shift in developer ergonomics and governance.

Traditional agent code tends to look like this:

const agent = new Agent({ model });
agent.addTool(searchTool);
agent.addTool(githubTool);
agent.addSkill(prdSkill);
agent.addSubAgent(researcher);
agent.addApprovalPolicy(approvalPolicy);
agent.addMemory(memoryStore);
agent.run();

A filesystem-first equivalent might look like this:

agent/
  agent.yaml
  instructions.md

  skills/
    write-prd.md
    code-review.md
    summarize-research.md

  tools/
    github.tool.ts
    jira.tool.ts
    search.tool.ts

  subagents/
    researcher/
      agent.yaml
      instructions.md
      tools/
      skills/

    editor/
      agent.yaml
      instructions.md
      skills/

  approvals/
    production-write.yaml
    external-email.yaml

  schedules/
    daily-briefing.yaml

  evals/
    prd-quality.yaml
    code-review-regression.yaml

The runtime’s job is to read this repository and answer four questions:

  1. What is the agent allowed to do? Instructions, tools, scopes, approval policies, and identity bindings.
  2. What capabilities does it have? Skills, tools, subagents, memory, retrieval indexes, and domain knowledge.
  3. How does it execute? Runtime backend, workflow engine, sandbox, scheduling, retries, checkpoints, and human-in-the-loop pauses.
  4. How do we know it works? Evals, traces, golden tasks, red-team suites, regression gates, and release policy.

2.1 Why the filesystem is a good interface

  • Inspectable. A buyer, security reviewer, or delivery lead can see what the agent is instructed to do without reverse-engineering the app.
  • Versionable. Git diffs show changes to skills, tools, permissions, and evals. That makes agent evolution reviewable.
  • Composable. Capabilities can be copied, templated, forked, vendored, and packaged.
  • Runtime-independent. The agent definition can outlive any single cloud provider or orchestration library.
  • Readable by non-code contributors. Product, design, compliance, domain experts, and client stakeholders can edit markdown skills, policies, examples, and acceptance tests.

2.2 The deeper analogy: containers and Kubernetes

The long-term opportunity is not “make prompts files.” It is an agent specification that can be compiled to multiple runtimes. Containers separated application packaging from host machines. Kubernetes separated desired state from implementation. A filesystem agent spec could separate agent definition from cloud runtime.

In that world, a consulting team could maintain an internal library of skills, tool packs for enterprise systems, subagent archetypes for research, planning, coding, testing, and delivery, evaluation suites, approval templates, and cloud-specific deployment adapters. The repository becomes the durable artifact. The cloud runtime becomes replaceable infrastructure.

3. Vercel Eve

Eve’s strongest idea is not that it can call tools or coordinate subagents. Those are table stakes. Its strongest idea is that the agent is a directory, and the platform supplies the production harness.

A representative Eve-style layout is:

agent/
  instructions.md
  config.ts

  tools/
    search.ts
    github.ts
    jira.ts

  skills/
    code-review.md
    write-prd.md
    summarize-research.md

  subagents/
    researcher/
    planner/
    implementer/

  approvals/
  schedules/
  channels/

Vercel’s documentation describes Eve as compiling this agent directory into an application that runs on Vercel Functions, using Vercel’s associated platform services for durability, sandboxing, identity, model routing, and observability.1

3.1 What Eve gives you

ExecutionVercel FunctionsDeploys naturally into Vercel’s serverless runtime.
DurabilityVercel WorkflowsSupports long-running work, waits, resumptions, and checkpoints.
SandboxingVercel SandboxEnables isolated execution for risky code/tool steps.

CredentialsVercel ConnectManages OAuth and API-key flows.
ObservabilityVercel ObservabilityTracks runs, tokens, latency, and performance.

3.2 The technical fellow read

Eve is best understood as an agent harness, not just an agent framework. Most frameworks give you primitives for reasoning, tools, and orchestration. Eve tries to give you the missing production layer: durable execution, approvals, deployment, channels, schedules, sandboxing, and observability.

That is the right emphasis. The failure mode of many enterprise agent experiments is not that the LLM cannot choose a tool. It is that the surrounding system cannot guarantee identity boundaries, recover from partial failure, resume after approval, explain what happened, evaluate regressions, or support release discipline.

3.3 The trade-off

Eve’s simplicity is achieved by leaning into Vercel’s platform. That is rational for Vercel and attractive for greenfield web teams. It is less clean for enterprises standardized on Azure, Microsoft 365, Entra ID, private networking, internal compliance gates, and existing cloud operations practices. Eve gives a superb authoring model, but its runtime gravity is Vercel.

4. The Azure agent environment

Microsoft’s agent ecosystem is powerful but distributed across several products and concepts. The core pieces are:

NeedAzure / Microsoft answer
Agent SDK and orchestrationMicrosoft Agent Framework
Managed agent hostingMicrosoft Foundry Agent Service
ModelsAzure OpenAI and Foundry models
Enterprise identityMicrosoft Entra ID and managed identities
SecretsAzure Key Vault
Long-running workflowsAzure Durable Functions or workflow patterns hosted around Foundry agents
Sandboxed executionAzure Container Apps Jobs, AKS, or isolated containers
Retrieval and enterprise searchAzure AI Search, Microsoft Graph, Fabric, and domain-specific stores
ObservabilityApplication Insights, Azure Monitor, OpenTelemetry, and Foundry tracing/evals
GovernanceAzure Policy, RBAC, private networking, content safety, model management, data residency controls

4.1 Microsoft Agent Framework

Microsoft Agent Framework is the authoring/orchestration layer. Microsoft describes it as a convergence of AutoGen and Semantic Kernel. The direction is obvious: make agent building approachable while retaining enterprise concerns such as state, type safety, middleware, telemetry, workflow orchestration, model support, and integration with Microsoft’s cloud platform.2

In practice, that means Microsoft’s framework is code-first. You define agents and workflows in Python or .NET, wire tools and middleware, then deploy or host the result using Azure/Foundry primitives.

4.2 Microsoft Foundry Agent Service

Foundry Agent Service is the managed platform layer. It supports prompt agents and hosted agents, managed endpoints, scaling, versioning, identity, tracing, evaluation, publishing, and enterprise security features such as Entra identity, private networking, and RBAC.3

This is not Eve, but it is arguably the stronger enterprise substrate. It fits organizations that already rely on Microsoft identity, Microsoft 365, Azure networking, Azure governance, and regulated deployment processes.

4.3 The gap

The missing abstraction is the filesystem. Microsoft has a strong runtime and enterprise cloud substrate, but the developer interface is still mostly SDK-first and portal-first. Eve’s advantage is not that Vercel has more enterprise primitives than Azure. It is that Eve gives developers a highly legible way to package an agent as a product.

5. Designing a filesystem agent framework on Azure

An Azure-native filesystem agent framework should not mimic Eve line by line. It should preserve the important idea—the agent repository is the product boundary—while compiling to Microsoft’s stronger enterprise runtime.

5.2 Compiler architecture

The framework should have a compiler that converts the repository into a normalized agent graph:

agent repository
  → parser
  → schema validator
  → capability graph
  → policy checker
  → Microsoft Agent Framework generator
  → Azure deployment adapter

The normalized graph is the important internal artifact. It should contain agents, subagents, skills, tools, tool schemas, approval gates, workflow edges, memory bindings, eval suites, deployment targets, and permissions. Once that exists, Azure becomes a target rather than the definition itself.

5.3 Azure mapping

5.5 The runtime behavior

At runtime, the agent should enforce policy outside the model. The model may decide it wants to send an email, update a ticket, or execute code. The runtime should decide whether that is allowed, whether approval is required, which identity is used, which scopes apply, how the action is logged, and whether the trace needs redaction. The filesystem spec is the declared intent; Azure controls are the enforcement mechanism.

6. How the Azure version would differ from Eve

DimensionVercel EveAzure-native filesystem framework
Primary interfaceFilesystem-firstFilesystem-first spec plus compiler
RuntimeVercel Functions and Vercel platform servicesFoundry Agent Service, Microsoft Agent Framework, Durable Functions, Container Apps, AKS, or Functions
Language centerTypeScript / JavaScriptPython and .NET first, with TypeScript possible later
IdentityVercel Connect and app credentialsEntra ID, managed identity, RBAC, conditional access
Enterprise dataConnectors and custom toolsMicrosoft Graph, Azure AI Search, Fabric, SharePoint, Teams, Dataverse, client VNET resources

Deployment postureOptimized for Vercel-native app teamsOptimized for enterprise Azure estates and regulated clients
Strategic riskElegant but platform-coupledMore complex but more governable and client-aligned

The Azure version should be less magical and more explicit. Eve’s advantage is developer elegance. Azure’s advantage is enterprise control. A serious Azure filesystem framework should not hide that trade-off. It should make the agent legible at the repository level and enforceable at the cloud level.

7. Why build it

7.1 For enterprise adoption

Large clients do not merely need agents that work in demos. They need agents that can be reviewed, approved, deployed, monitored, and retired. They need identity boundaries, networking boundaries, data boundaries, and release gates. Azure already has many of those primitives. The missing piece is a clean authoring and packaging layer.

7.2 For forward deployed engineering

A filesystem framework is especially useful for forward deployed engineering because it turns engagements into reusable assets. A team can walk into a client engagement with a reference repo, client-specific skills, approved tool packs, prebuilt eval suites, security policy templates, deployment adapters, and a demo-to-production path that maps to Azure governance.

This is better than a pile of notebooks, prompts, and ad hoc integrations. It is a repeatable productized delivery system.

7.3 For auditability

Auditability is not just logs. It is the ability to answer: What was the agent instructed to do? What tools could it call? Which data sources could it access? Which actions required approval? Which evals were required before release? Which version was running when a decision was made?

A filesystem spec plus Git plus Azure observability can answer these questions cleanly.

7.4 For portability

Even if the first compiler targets Azure, the agent definition should not be inseparable from Azure. The right abstraction is:

agent repository → normalized agent graph → runtime adapter → deployed agent

Today’s adapter might target Foundry Agent Service. Tomorrow’s adapter might target Vercel, AWS, on-prem Kubernetes, or a local developer runtime. That is the strategic value: avoid confusing the agent specification with the agent host.

8. Implementation plan

Phase 1: Define the spec

Create a small, opinionated schema for the core objects: Agent, Skill, Tool, Subagent, Workflow, ApprovalPolicy, MemoryBinding, EvalSuite, and DeploymentTarget. Keep the schema boring. YAML for configuration, Markdown for instruction-bearing artifacts, JSON Schema for tool inputs/outputs, Python or .NET for handlers.

Phase 2: Build a local compiler

The first compiler should not deploy anything. It should load the repository, validate schemas, construct a capability graph, generate a human-readable manifest, run static checks, and run local evals.

atlas validate ./agent
atlas graph ./agent --output manifest.json
atlas eval ./agent --suite evals/regression.yaml

Phase 3: Generate Microsoft Agent Framework code

Generate a minimal Python and/or .NET project from the normalized graph. Do not ask every project team to hand-wire the same orchestration code. Generate it.

atlas generate ./agent --target microsoft-agent-framework --language python

The generated project should include agent definitions, tool registration, subagent registration, workflow definitions, middleware for policy checks, OpenTelemetry instrumentation, eval hooks, and deployment manifests.

Phase 4: Add Azure deployment adapters

Support two deployment paths: a Foundry hosted agent path that packages generated code as a hosted agent, and an Azure-native service path that deploys to Azure Container Apps or AKS with Durable Functions for long-running workflows and App Insights for telemetry.

atlas deploy ./agent --target foundry --environment prod

Phase 5: Add governance gates

Add gates that fail the build if a high-risk tool lacks an approval policy, a tool uses a raw secret instead of Key Vault, a data source crosses a declared residency boundary, an external-write tool lacks trace redaction, eval coverage is missing for critical skills, a subagent has broader permissions than its parent, or a workflow lacks retry and failure behavior.

Phase 6: Package reusable packs

packs/
  azure-devops-tools/
  jira-tools/
  service-now-tools/
  product-strategy-skills/
  secure-code-review-skills/
  fde-delivery-evals/
  regulated-industry-policies/

These should be versioned and internally certified. Client teams should consume them the way they consume infrastructure modules or internal libraries.

9. Risks and hard questions

9.1 The filesystem can become a junk drawer

A folder structure is not architecture by itself. Without strict schemas, naming conventions, ownership, and tests, a filesystem-first framework degenerates into prompt sprawl with nicer folders.

9.2 Declarative configuration can hide real complexity

Declarative systems are powerful until debugging requires understanding the compiler, generated code, cloud runtime, and model behavior simultaneously. The framework must provide excellent introspection: generated manifests, dependency graphs, dry runs, trace replay, and explanations of why a tool was available.

9.3 Human-readable does not mean safe

Markdown instructions are readable, but they are not enforceable. Policies must be enforced in middleware, identity, network controls, tool wrappers, and approval gates. A skill file saying “do not export customer data” is not a control.

9.4 Runtime portability is harder than spec portability

A generic agent spec can be portable. Production semantics are not. Durable execution, identity, sandboxing, and observability differ across platforms. The compiler should expose those differences rather than pretending they do not exist.

9.5 Azure complexity is real

Azure gives you enterprise primitives, but it also gives you surface area: subscriptions, resource groups, VNets, managed identities, private endpoints, RBAC, Key Vault, App Insights, and deployment templates. A filesystem agent framework must simplify this without hiding the controls that enterprises actually need.

10. Recommended position

Do not position an Azure filesystem framework as “Eve for Azure.” That undersells it and accepts Vercel’s framing. The stronger position is:

An Azure-native filesystem agent framework is an enterprise agent specification and compiler. It gives teams Eve-like authoring ergonomics while targeting Microsoft’s stronger enterprise runtime: Foundry, Entra, Azure networking, Durable Functions, OpenTelemetry, App Insights, Key Vault, and Microsoft 365 distribution.

For a forward deployed engineering organization, this becomes a delivery accelerant. It gives teams a standard way to package agent behavior, client-specific knowledge, approved tools, evals, and deployment policy. It also gives clients something they can inspect and govern.

The agent repository is the product boundary. The runtime is an adapter.

That is the idea worth carrying forward from Eve. Not Vercel Functions. Not a particular folder name. The important move is to make the agent legible, portable, reviewable, and deployable by turning its behavior into a structured repository.
