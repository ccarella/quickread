The Agent as a Directory

Designing a cloud-agnostic filesystem-based agent framework inspired by Vercel Eve, deployable to Azure, Google Cloud, AWS, or Kubernetes.

Prepared July 2026. Written as an architectural report for builders who want agents to become legible, testable, portable software rather than dashboard objects.

1. Executive summary

The most important idea in Vercel Eve is not that it runs on Vercel. It is that an agent can be expressed as a directory.

A directory has files. Files are durable interfaces. They are versionable, inspectable, diffable, reviewable, testable, vendorable, forkable, and understandable by humans and coding agents. That makes the filesystem an unusually good surface for agent software, because agents are partly natural-language systems and therefore need more governance, not less.

Vercel describes Eve as a filesystem-first framework for durable backend AI agents. An agent lives under an agent/ directory; Eve discovers files such as instructions.md, agent.ts, tools/, skills/, subagents/, channels/, and schedules/, then compiles that directory into an app running on Vercel infrastructure.

The bigger opportunity: separate the agent authoring model from the cloud execution substrate. Build an Eve-like developer experience, but make the source format portable across Azure, Google Cloud, AWS, local machines, and Kubernetes.

The target product should feel like this:

The winning design principle is simple: the filesystem is the product surface; the cloud is an implementation detail.

2. Why filesystem-first is the right primitive

Most agent frameworks start from runtime objects: agents, tools, graphs, routers, memories, evaluators. That is natural for library authors, but it is not how enduring software systems are governed. Real systems are governed through files, pull requests, build artifacts, deployment manifests, test fixtures, logs, and policy.

A filesystem-first agent framework makes behavior concrete. instructions.md is not a blob hidden in a SaaS dashboard. A tool is not a row in a control plane. A skill is not a prompt fragment lost in a database. An eval is not a spreadsheet someone forgot to update. A schedule is not some cron thing in production. All of them are source.

This also gives humans and coding agents a clean collaboration model. A human can say, “Add a skill for handling EU refunds.” A coding agent can create skills/eu-refunds.md, add an eval under evals/, update policy, and open a pull request. That is better than clicking through an agent console until behavior changes.

Provocation: agent frameworks should look less like LangChain graphs and more like Next.js, Rails, and Unix. The directory is the app.

3. The core architecture

A cloud-agnostic filesystem agent framework should have three layers:

Authoring Layer
  Files, conventions, schemas, local dev server, tests, evals

Compilation Layer
  Static analysis, tool discovery, prompt assembly, policy checks, manifest generation

Runtime Layer
  Durable runs, model calls, tool execution, sandboxing, memory, approvals, tracing, deployment adapters

Begin with the authoring layer. The authoring layer is what developers fall in love with. The runtime is what enterprises require. The compiler is what lets you satisfy both.

The portable manifest

The source directory should compile into a deterministic, machine-friendly manifest:

name: customer_refund_agent
version: 0.4.2
model:
  provider: openai
  model: gpt-5.4
instructions:
  file: instructions.md
skills:
  - id: refund_policy
    file: skills/refund-policy.md
tools:
  - name: query_order
    source: tools/query_order.ts
    risk: read
  - name: issue_refund
    source: tools/issue_refund.ts
    risk: write
    approval_required: true
runtime:
  durability: required
  sandbox: required
  max_run_time: 8h
observability:
  traces: opentelemetry

The compiler should produce cloud-specific deployment artifacts from that manifest. For Azure, it might generate a Foundry Hosted Agent deployment, a Durable Functions orchestration, or an Azure Container Apps job. For Google Cloud, it might target ADK, Agent Platform Runtime, Workflows, Cloud Run, or GKE. For AWS, it might target Bedrock AgentCore Runtime, Step Functions, Lambda, ECS/Fargate, or EKS.

The source remains stable. The deployment substrate changes.

4. Runtime design decisions

Durability first

Agents wait on model calls, flaky APIs, human approvals, retries, and rate limits. The agent loop itself should be durable. Model calls, tool calls, approval waits, and memory writes should be durable steps.

Tools as ordinary code

Tool authoring should feel boring, but tool execution should be highly controlled. Every tool needs schema, description, risk classification, auth requirements, timeout, retry policy, idempotency strategy, approval policy, network policy, and trace redaction.

Sandbox by default

Any agent that writes files, executes code, browses, installs packages, or processes untrusted documents needs sandboxing as a baseline feature.

Memory as event log

Start with event-sourced memory before semantic memory. Every memory should know where it came from, when it was observed, whether it was inferred, when it expires, and which agents may read it.

Observability for cognition

Agent traces must answer: what did the agent believe it was doing, which instructions were active, which tools were chosen, what evidence was used, and where cost and latency accumulated?

Evals compile with the agent

An agent without evals should deploy with a warning. A high-risk agent without evals should fail deployment. Evals should run locally, in CI, and against deployed environments.

5. Cloud adapter strategy

The right abstraction is not “write once, run identically everywhere.” That is usually a lie. The right abstraction is: write once, preserve semantics, exploit the target.

TargetBest fitLikely mappings
AzureEnterprise identity, Microsoft 365, Teams, Entra ID, private networking, governance.Foundry Hosted Agents, Durable Functions, Container Apps, Blob Storage, Cosmos DB, Azure AI Search, Key Vault, Azure Monitor.
Google CloudModel-heavy, data-heavy, BigQuery-heavy workloads; ADK and Agent Platform Runtime.ADK, Agent Platform Runtime, Cloud Run, Workflows, Cloud Storage, Firestore, BigQuery, Cloud Trace.
AWSBroadest primitive set, Step Functions, Bedrock, mature IAM, production controls.Bedrock AgentCore Runtime, Step Functions, Lambda, ECS/Fargate, S3, DynamoDB, OpenSearch, CloudWatch.
KubernetesMaximum portability and control.Temporal, Restate, DBOS, Postgres event log, object storage, OpenTelemetry collector.

Do not hide cloud differences. Isolate them behind adapters. The Azure, AWS, Google, local, and Kubernetes adapters should be replaceable packages. The framework should survive any one provider becoming unattractive.

7. Opinionated design principles

  1. The agent directory is the API. If a feature exists only in a cloud console, it is not part of the framework yet.
  2. The compiler is the control plane. It should validate, type-check, assemble, package, and explain the agent.
  3. Every run is an artifact. Inputs, instructions, skills, model calls, tool calls, approvals, memory operations, costs, latency, and final outputs should be inspectable.
  4. Dangerous tools require explicit policy. Anything that sends email, moves money, updates records, deletes files, publishes content, or changes permissions needs risk metadata and approval behavior.
  5. Skills are source code. Natural-language instructions are behavioral code. Treat them with review, tests, versioning, and rollback.

8. Building with the framework 101: a PDF summarization agent

The best way to understand the framework is to build something ordinary: an agent that summarizes PDFs. It receives a PDF, extracts text, chunks content, summarizes each chunk, synthesizes a final summary, saves an artifact, and records a trace.

agent init pdf-summarizer
cd pdf-summarizer
pdf-summarizer/
  agent/
    instructions.md
    agent.yaml
    tools/
    skills/
    evals/
    fixtures/
  .agent/
    runs/
    compiled/

Manifest

name: pdf_summarizer
version: 0.1.0
model:
  provider: openai
  model: gpt-5.4
  temperature: 0.2
runtime:
  durability: required
  sandbox: required
  max_run_time: 30m
input:
  type: file
  accepted_mime_types:
    - application/pdf
policy:
  allow_external_network: false
  allow_persistent_memory: false
  require_page_references: true

Instructions

# PDF Summarizer Agent

You summarize PDF documents for a technical reader.

Return Markdown with these sections:
1. Executive Summary
2. Document Structure
3. Key Claims
4. Important Details
5. Risks, Gaps, or Ambiguities
6. Actionable Takeaways
7. Page-Referenced Notes

Rules:
- Use only the uploaded PDF.
- Do not invent facts.
- Include page references when available.
- If extraction quality is poor, say so directly.

Tools

Use small, boring tools: extract_pdf_text, chunk_document, and write_artifact. The extraction tool does not summarize. The chunking tool does not interpret. The artifact writer does not call models. Narrow tools make traces readable.

tool: extract_pdf_text
input: filePath
output: pages[], extractionQuality, warnings
risk: read

tool: chunk_document
input: pages[], targetTokens
output: chunks[]
risk: read

tool: write_artifact
input: filename, content
output: artifactPath
risk: write

Run and inspect

agent compile
agent run agent/fixtures/sample.pdf
agent trace latest
agent eval

The local runtime should create a run directory containing input, messages, tool calls, trace, artifacts, and final output. This is the first payoff: the agent does not merely answer; it leaves behind an auditable run.

9. Building with the framework 201: reading PDFs from Azure Blob Storage

The 101 example used a local file. Production documents usually arrive through object storage. The 201 version reads a PDF from Azure Blob Storage, downloads it into the sandbox workspace, summarizes it, and uploads the summary back to Blob Storage.

PDF lands in Azure Blob Storage
        ↓
Blob-created event fires
        ↓
Agent run starts
        ↓
Agent downloads PDF into sandbox workspace
        ↓
Agent extracts, chunks, summarizes
        ↓
Summary uploads to output container

Manifest upgrade

name: pdf_summarizer
version: 0.2.0
input:
  type: azure_blob
  accepted_mime_types:
    - application/pdf
azure_storage:
  account: ${AZURE_STORAGE_ACCOUNT}
  input_container: incoming-pdfs
  output_container: summarized-pdfs
  auth: managed_identity
policy:
  allowed_blob_containers:
    read:
      - incoming-pdfs
    write:
      - summarized-pdfs

The key design decision: cloud input is normalized into a local workspace file. The existing extract_pdf_text tool still receives a local filePath. It should not know where the file came from.

Azure boundary tools

tool: download_azure_blob
input: accountName, containerName, blobName
output: filePath, originalBlobName, contentType, sizeBytes
risk: read
policy: may read only incoming-pdfs and application/pdf

tool: upload_azure_blob
input: accountName, containerName, blobName, localFilePath, contentType
output: blobUrl
risk: write
policy: may write only summarized-pdfs from artifact directory

Use managed identity rather than account keys. Assign the runtime identity Storage Blob Data Reader on the input container and Storage Blob Data Contributor on the output container. Do not grant broad storage account access just because this is “only summarization.” Small agents become exfiltration tools when storage access is sloppy.

Workflow

workflow summarize_pdf_from_azure:
  downloaded = download_azure_blob(input)
  extracted = extract_pdf_text(downloaded.filePath)
  chunks = chunk_document(extracted.pages)
  chunkSummaries = summarize each chunk
  final = synthesize chunkSummaries
  artifact = write_artifact(summary.md, final)
  upload_azure_blob(output container, artifact.path)

Event channel

type: azure_blob_event
name: summarize_pdf_when_blob_created
container: incoming-pdfs
events:
  - Microsoft.Storage.BlobCreated
filters:
  suffix:
    - .pdf
workflow: summarize_pdf_from_azure

The source file declares intent. The Azure adapter chooses whether to implement it with Event Grid, a Blob-triggered Function, Durable Functions, or another event handler.

10. Building with the framework 301: 100 PDFs and subagents

The 301 version summarizes a corpus of 100 PDFs. That is no longer just document processing. It is coordination: discovery, batching, concurrency limits, partial failure handling, subagent assignment, intermediate artifacts, corpus-level synthesis, and provenance.

List 100 PDFs from Azure Blob Storage
        ↓
Create a corpus run
        ↓
Spawn document subagents
        ↓
Each subagent summarizes one PDF
        ↓
Parent waits for structured results
        ↓
Batch synthesizers compress groups
        ↓
Corpus synthesizer writes overarching summary

Directory upgrade

agent/
  workflows/
    summarize_corpus_from_azure.workflow.ts
    summarize_single_pdf.workflow.ts
    synthesize_corpus.workflow.ts
  subagents/
    document_summarizer/
      instructions.md
    batch_synthesizer/
      instructions.md
    corpus_synthesizer/
      instructions.md
  schemas/
    document_summary.schema.json
    batch_summary.schema.json
    corpus_summary.schema.json
  tools/
    list_azure_blobs.ts
    download_azure_blob.ts
    upload_azure_blob.ts
    extract_pdf_text.ts
    chunk_document.ts
    write_artifact.ts

Manifest upgrade

name: pdf_corpus_summarizer
version: 0.3.0
input:
  type: azure_blob_prefix
corpus:
  max_documents: 100
  discovery:
    mode: prefix
  document_concurrency: 8
  max_document_failures: 10
  batch_size: 10
  batch_concurrency: 3
  synthesis_mode: hierarchical
subagents:
  document_summarizer:
    output_schema: schemas/document_summary.schema.json
  batch_synthesizer:
    output_schema: schemas/batch_summary.schema.json
  corpus_synthesizer:
    output_schema: schemas/corpus_summary.schema.json

The parent agent should not read all 100 PDFs directly. The parent coordinates. Document subagents process documents. Batch synthesizers compress groups. The corpus synthesizer works from structured summaries, not raw PDFs.

Typed hierarchical compression

PDF → DocumentSummary
DocumentSummary[] → BatchSummary
BatchSummary[] → CorpusSummary

This is the critical 301 idea. Reliable multi-agent systems are not swarms of chatty agents. They are typed compression pipelines with clear role boundaries.

Parent workflow

workflow summarize_corpus_from_azure:
  blobs = list_azure_blobs(prefix, maxDocuments=100)
  manifest = write corpus-manifest.json
  documentResults = fan out document_summarizer with concurrency 8
  normalize failures using Promise.allSettled semantics
  if failedCount > maxDocumentFailures: fail safely
  batches = group documentResults by 10
  batchResults = fan out batch_synthesizer with concurrency 3
  corpusSummary = corpus_synthesizer(batchResults, compact document summaries)
  write corpus-summary.md and corpus-summary.json
  upload artifacts to summarized-pdfs/corpus-runs/{runId}/

Partial failure policy

Fail the corpus run if discovery fails.
Fail the corpus run if too many documents fail.
Do not fail the corpus run because one document fails.
Always disclose coverage in the final summary.

The READY file pattern

Do not start a corpus job on every individual BlobCreated event. A corpus needs a boundary. Use a sentinel file:

incoming-pdfs/diligence/acme/_READY.json
{
  corpusId: acme-diligence-2026-07,
  prefix: diligence/acme/,
  expectedDocuments: 100,
  createdBy: upstream-diligence-export,
  createdAt: 2026-07-02T21:00:00Z
}

When _READY.json appears, the agent summarizes the prefix. This prevents starting a corpus summary while only 37 of 100 PDFs have arrived.

Final report coverage

# Corpus Summary

## Coverage
- Input container: incoming-pdfs
- Prefix: diligence/acme/
- Documents discovered: 100
- Documents summarized: 96
- Partial summaries: 1
- Failed documents: 3
- Synthesis mode: hierarchical
- Batch size: 10

This summary is based on 97 usable document summaries. Three documents failed and are listed in the document index.

The first paragraph of the final report should tell the reader how much of the corpus was actually analyzed. Anything else invites overconfidence.

What 301 teaches

A normal script can summarize one PDF. A decent cloud function can summarize a PDF from Blob Storage. But a filesystem-based agent framework can give you source-controlled roles, subagent directories, typed inter-agent contracts, durable fan-out/fan-in, bounded concurrency, partial failure tolerance, hierarchical synthesis, auditable traces, corpus manifests, human-readable and machine-readable artifacts, and portable cloud targets.

The product philosophy still holds: the agent is a directory. The subagent is a subdirectory. The run is an artifact tree. The corpus is a manifest. The cloud is a target. The framework is the compiler and runtime contract between them.
